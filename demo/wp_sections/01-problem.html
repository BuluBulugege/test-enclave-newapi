1 · 问题与信任模型

当你把一段 prompt 发给任意一个普通 AI 网关时，请求虽然走 HTTPS 加密传输，但网关进程会在服务端终止 TLS、拿到明文，然后才转发给上游模型。也就是说，在那一刻，运营方的代码能看见你发的每一个字，也能把它写进磁盘、数据库或日志。你之所以敢发，靠的只是一句「我们不记录」——一句无法验证的口头承诺。

本方案要解决的，正是这个「信任」问题：不是让你换一个「更值得相信」的运营方，而是把「相信」这件事从人的品德迁移到可用密码学独立验证的数学事实上。核心手段是把负责转发的 relay-core 放进 Intel SGX enclave（Gramine-SGX 1.9 + DCAP 远程认证）里运行，让「服务器看不到、也存不下你的正文」成为一条任何人都能亲自核验的结论。

1.2 「无日志」为什么不可信

「我们不记录你的数据」这类声明，在传统架构下无法转化为可核验的证据。原因有四：

  • 代码不可见：你看到的是文档和承诺，看不到线上实际运行的二进制。开源仓库不等于线上部署的就是同一份代码。
  • 可随时被改：即便某一刻代码干净，运营方（或被入侵者、或云厂商 root）可以在任意时刻热更新、注入抓包、加一行落盘逻辑，而你毫无察觉。
  • 特权方在链路上：宿主 OS、Hypervisor、云平台运维都能读进程内存与磁盘，「不记录」对这些特权层没有任何强制约束。
  • 无法事后证伪：即使出了泄露事件，你也很难证明是网关记录了明文——责任与举证都对用户不利。

结论是：「无日志」在传统架构里只能被承诺，不能被证明。它的可信度上限等于「你有多相信这家运营方及其所有特权依赖方」。

1.3 信任模型：把「相信」变成「可验证」

本方案不要求你相信运营方的品德，而是把结论建立在三块可以独立核验的密码学基石上。任意一环对不上，客户端在发送 prompt 之前就会当场判定失败并拒绝发送。

这三块基石的作用分别是：

  • MRENCLAVE = 代码 + manifest 的度量（当前上线值 4aa951d16a0c237605f032cd480095b65be1f485e9f7a959a16f38a80428a445）。它对 enclave 内的代码与配置做哈希，改动任意一字节，指纹即完全改变——等价于 Docker digest / git commit / 前端 SRI。
  • Quote（远程认证凭证）由 CPU 出具、带 Intel 签名，通过 DCAP 验证签名链，证明「这是一块真的 SGX 芯片，此刻正在运行 MRENCLAVE 所标识的这份确切代码」。伪造不了。
  • RA-TLS 通道绑定把 report_data 固定为 SHA-512(TLS SubjectPublicKeyInfo)，即公钥指纹被写进被 Intel 签名的报告里。它保证「被认证的那个 enclave，就是你此刻正在通话的这个端点」，而不是被中间人转发或重放来的另一台。

三者相乘产生传递性证明：代码开源可审计 ⇒ 审计确认它没有把正文写盘 / 入库 / 记日志的代码路径、且只连编译期写死的官方上游 ⇒ Quote 证明线上跑的就是这份被审计的代码 ⇒ 因此它在结构上「做不到」偷看。任何篡改都会改变 MRENCLAVE，验证阶段立刻暴露。

1.5 信任边界（TCB）与威胁模型

本方案明确划定了可信计算基（TCB）的边界：

  • 可信（TCB 内）：Intel SGX CPU 与其认证机制、enclave 内运行的 relay-core 代码、以及被 MRENCLAVE 度量的 Gramine manifest。目的地 URL 与各上游的鉴权画像都编译进 enclave 并被度量，运营方无法改动。
  • 不可信（TCB 外）：作为控制面的 new-api、宿主操作系统、Hypervisor、云平台运维、磁盘 / 数据库 / 日志系统。它们都被默认视为敌手：只能看到路由所需的 {model, token}，看不到 prompt 与回答正文。
  • 官方上游：OpenAI（Authorization: Bearer）、OpenRouter（Bearer + HTTP-Referer / X-Title）、Anthropic（x-api-key + anthropic-version）、Gemini / AI Studio（x-goog-api-key）。enclave 用编译期写死的官方地址加严格 TLS 直连，凭据在 enclave 内注入。

本方案不声称防护的范围（如实说明）：上游提供商拿到的仍是你的 prompt 明文——本方案保护的是「本网关及其宿主环境看不到你的数据」，而非上游模型服务方；SGX 侧信道等硬件层面的攻击不在本文讨论范围；单纯拿到 Quote 还不足以完全杜绝伪造，必须再做 DCAP 签名链验证（relay-verify -dcap-verify）。后续章节将逐层展开每一环的实现与验证方法。

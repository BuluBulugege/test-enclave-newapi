3 · MRENCLAVE 代码度量

要证明"服务器跑的就是这份不落盘的审计代码"，首先得有一个无法伪造的代码身份。在 SGX 里，这个身份就是 MRENCLAVE——一个 32 字节（SHA-256）的度量值。它不是由运营方声明的，而是在 enclave 装载的一瞬间，由 CPU 硬件逐页计算出来的。运营方无法在不改变这个指纹的前提下，往 enclave 里塞进任何一行"偷偷写盘"的代码。

3.1 逐页度量：EADD → EEXTEND → EINIT

Gramine-SGX 1.9 把 relay-core 及其运行时打包成 enclave 镜像后，交给 CPU 装载。装载过程中，CPU 对每一页内存执行两条指令：EADD 把页加入 enclave，EEXTEND 把该页按 256 字节为单位喂进一个持续累积的 SHA-256 中。喂进哈希的不只是页里的字节，还包括这一页的虚拟地址（在 enclave 里的偏移）和安全属性（SECINFO：读/写/执行权限、页类型）。所有页处理完后，EINIT 冻结这个累积值——它就是 MRENCLAVE。

  • 代码页（.text）——可执行指令，即 relay-core 的全部业务逻辑。
  • 数据页（.data / .rodata）——初始化数据与常量，例如编译写死的官方上游 URL。
  • 堆 / 栈 / 控制结构（SSA、TCS）的页布局——不是内容，而是"哪一页放在哪个地址、给什么权限"。
  • manifest（relay-core.manifest）——Gramine 的信任配置：入口点、环境变量、可信文件哈希、挂载策略。它同样进入度量，所以改配置也会改指纹。

因为地址与权限都进了哈希，即使两份 enclave 的代码字节完全一样，只要页布局不同，MRENCLAVE 也会不同。这正是"逐页度量"比"只哈希一个二进制文件"更强的地方：它锁定的是加载进内存后的完整状态，而不只是磁盘上的文件。

3.2 雪崩效应：改一字节，指纹全变

SHA-256 具有雪崩效应（avalanche effect）：输入哪怕只翻转 1 个 bit，输出也会有约一半的 bit 随之翻转，且新旧值之间毫无可预测的关系。对 enclave 而言这意味着：运营方想在代码里插一句"把 prompt 写进日志"，或在 manifest 里悄悄挂载一个可写目录，得到的都将是一个与 pin 值毫不相干的新指纹——客户端在验证第 2 步（度量比对）会立刻变红，请求根本发不出去。

3.3 类比：你早就在用的"内容指纹"

MRENCLAVE 并不神秘——它和你日常依赖的多种"内容寻址"指纹是同一类东西：都用密码学哈希把一坨字节压成一个短标识，凭它就能判断"这份东西是不是那份确切的东西"。区别只在于 MRENCLAVE 由 CPU 在硬件里计算、并被 Intel 的远程认证 quote 签名背书，因此运营方连"谎报指纹"都做不到。

机制度量对象摘要 / 长度类比要点
MRENCLAVE（SGX） 代码页 + 数据页 + 页布局 + manifest SHA-256 / 32 字节 硬件度量、被 quote 签名，改一字节即失配
Docker 镜像 digest 镜像 config + 各 layer 内容 SHA-256（sha256:…） 内容寻址，pull 时按 digest 校验
git commit id 提交树 + 父提交 + 作者/时间元数据 SHA-1 / SHA-256 改任意一行 → 全新 commit id
SRI（前端子资源完整性） JS / CSS 资源字节 SHA-384（integrity=） 哈希对得上，浏览器才执行该脚本

把这三小节串起来：MRENCLAVE 由 CPU 逐页度量得到，被写进远程认证 quote 的 report body 并由 Intel 背书；客户端在验证时把它与自己 pin 的期望值 e5c301b5f4256e08…5bd9b36a 逐字节比对（见第 5 节验证流程的 Check 2）。而这个期望值之所以可信，是因为整套构建可复现——固定 Go 1.26.1 + Gramine 1.9，任何人独立重新编译都会得到同一个 MRENCLAVE。于是"运营方看不到你的 prompt"这句承诺，被压缩成了一个可以当场核对的 32 字节数字。

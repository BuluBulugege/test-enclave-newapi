一次 AI 请求经历了什么 · 怎么验证它没偷看你

连的是真实运行在 Intel SGX 上的 enclave。两个按钮，看懂：① 怎么验证这台是真 enclave；② 你的 prompt 在每一跳到底去了哪。

读我先 · 3 分钟搞懂这在干嘛

你把 prompt（可能含敏感数据）发给 AI 网关时，正常情况下运营方能看见、能记录你发的每一个字——你只能选择"相信"他们不这么做。这个演示把"相信"换成"可验证"：用硬件 + 密码学证明服务器看不到也存不下你的数据，并让你亲眼看到数据每一跳去了哪。

先认识 3 个词（用你熟悉的东西类比）

🔒
Enclave（机密飞地）
CPU 里一块硬件加密的隔离区。程序在里面跑，它的内存连服务器的 root 管理员、连云厂商都看不到明文。
类比：一个连房东都打不开的保险箱，你的程序在保险箱里运行。
📜
Quote（远程认证凭证）
CPU 出具的、带 Intel 签名的一句话：“我这块真芯片，此刻正在跑这段确切的代码”。
类比：像 HTTPS 证书，但由 CPU 硬件 + Intel 背书，运营方伪造不了。
🧬
MRENCLAVE（代码指纹）
那段代码 + 配置的哈希值。改动一个字节，指纹就完全不同。
类比：Docker 镜像 digest / git commit 号 / 前端的 SRI。
那"它没偷看你"到底是怎么被证明的？ 一条链子，缺一环就不成立：
  1. 跑在 enclave 里的代码是开源的，任何人可读、可重新编译得到同一个指纹（MRENCLAVE）。
  2. 审计这份代码 ⇒ 确认它根本没有把正文写盘/入库/记日志的代码，且只连官方上游。
  3. Quote 由 Intel 硬件签名，证明服务器上跑的就是这份代码（指纹对得上）。
  4. ⇒ 传递性地证明：这台服务器做不到偷看/留存你的 prompt——运营方想作恶就得改代码，一改指纹就变，你下面的"开始验证"当场会红。

①先验证：这台是不是真 enclave？

发任何 prompt 之前，客户端先做三项检查——把"相信运营方"换成"相信 Intel 硬件签名 + 你能自己重建的代码指纹"。

②看一次请求经历了什么

下面 4 个方框是数据流经的 4 个角色。实线框 = 可信（enclave / 官方上游 / 你自己）；虚线框 = 不可信（new-api 主体、宿主、云厂商）。整套设计的核心就是：让你的 prompt 永远不跨过虚线框。点"发送并可视化"，逐跳看每条边界上到底流过了什么数据——重点看第 2 跳（发给不可信 new-api 的那一跳里，有没有你的 prompt）。

你 (客户端)发 prompt
▶
SGX Enclave加密内存 · 不落盘
▶
new-api 控制面不可信 · 只见元数据
▶
OpenAI 官方api.openai.com
绿色边 = 加密 / 可信通道 红色虚线边 = 跨越"不可信"边界（这里只应有元数据，绝无 prompt）

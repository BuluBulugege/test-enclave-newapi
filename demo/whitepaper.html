0 · 全链路总览

一句话讲清整套系统：你的客户端在发送任何 prompt 之前，先用远程认证（DCAP + RA-TLS）验证一个跑在 Intel SGX enclave 里的 relay-core —— 确认它就是那份可复现编译、经过审计、不落盘 / 不入库 / 不记日志的代码（MRENCLAVE 对得上）—— 然后把明文 prompt 经加密可信通道交给 enclave，由 enclave 直连 官方上游（OpenAI / OpenRouter / Anthropic / Gemini）；而 new-api 只是不可信的控制面，只回答"用哪个渠道、扣多少 token"这类路由问题，全程看不到也存不下你的正文。

1 · 问题与信任模型

当你把一段 prompt 发给任意一个普通 AI 网关时，请求虽然走 HTTPS 加密传输，但网关进程会在服务端终止 TLS、拿到明文，然后才转发给上游模型。也就是说，在那一刻，运营方的代码能看见你发的每一个字，也能把它写进磁盘、数据库或日志。你之所以敢发，靠的只是一句「我们不记录」——一句无法验证的口头承诺。

本方案要解决的，正是这个「信任」问题：不是让你换一个「更值得相信」的运营方，而是把「相信」这件事从人的品德迁移到可用密码学独立验证的数学事实上。核心手段是把负责转发的 relay-core 放进 Intel SGX enclave（Gramine-SGX 1.9 + DCAP 远程认证）里运行，让「服务器看不到、也存不下你的正文」成为一条任何人都能亲自核验的结论。

1.1 普通 AI 网关 vs 本方案：谁能看到你的 prompt

下图对比两种架构下 prompt 的可见范围。关键差别在中间那一跳：传统网关自身就是明文处理点；本方案里承担转发的是 enclave，而作为控制面的 new-api 被明确划为不可信，跨越它的边界只有 {model, token} 这类路由元数据，永远不含 prompt 正文。

1.2 「无日志」为什么不可信

「我们不记录你的数据」这类声明，在传统架构下无法转化为可核验的证据。原因有四：

  • 代码不可见：你看到的是文档和承诺，看不到线上实际运行的二进制。开源仓库不等于线上部署的就是同一份代码。
  • 可随时被改：即便某一刻代码干净，运营方（或被入侵者、或云厂商 root）可以在任意时刻热更新、注入抓包、加一行落盘逻辑，而你毫无察觉。
  • 特权方在链路上：宿主 OS、Hypervisor、云平台运维都能读进程内存与磁盘，「不记录」对这些特权层没有任何强制约束。
  • 无法事后证伪：即使出了泄露事件，你也很难证明是网关记录了明文——责任与举证都对用户不利。

结论是：「无日志」在传统架构里只能被承诺，不能被证明。它的可信度上限等于「你有多相信这家运营方及其所有特权依赖方」。

1.3 信任模型：把「相信」变成「可验证」

本方案不要求你相信运营方的品德，而是把结论建立在三块可以独立核验的密码学基石上。任意一环对不上，客户端在发送 prompt 之前就会当场判定失败并拒绝发送。

这三块基石的作用分别是：

  • MRENCLAVE = 代码 + manifest 的度量（当前上线值 98ba342adb8092d60c940e75ce8e07036c96226595b03f16bb8d35a52a1872ee）。它对 enclave 内的代码与配置做哈希，改动任意一字节，指纹即完全改变——等价于 Docker digest / git commit / 前端 SRI。
  • Quote（远程认证凭证）由 CPU 出具、带 Intel 签名，通过 DCAP 验证签名链，证明「这是一块真的 SGX 芯片，此刻正在运行 MRENCLAVE 所标识的这份确切代码」。伪造不了。
  • RA-TLS 通道绑定把 report_data 固定为 SHA-512(TLS SubjectPublicKeyInfo)，即公钥指纹被写进被 Intel 签名的报告里。它保证「被认证的那个 enclave，就是你此刻正在通话的这个端点」，而不是被中间人转发或重放来的另一台。

三者相乘产生传递性证明：代码开源可审计 ⇒ 审计确认它没有把正文写盘 / 入库 / 记日志的代码路径、且只连编译期写死的官方上游 ⇒ Quote 证明线上跑的就是这份被审计的代码 ⇒ 因此它在结构上「做不到」偷看。任何篡改都会改变 MRENCLAVE，验证阶段立刻暴露。

1.5 信任边界（TCB）与威胁模型

本方案明确划定了可信计算基（TCB）的边界：

  • 可信（TCB 内）：Intel SGX CPU 与其认证机制、enclave 内运行的 relay-core 代码、以及被 MRENCLAVE 度量的 Gramine manifest。目的地 URL 与各上游的鉴权画像都编译进 enclave 并被度量，运营方无法改动。
  • 不可信（TCB 外）：作为控制面的 new-api、宿主操作系统、Hypervisor、云平台运维、磁盘 / 数据库 / 日志系统。它们都被默认视为敌手：只能看到路由所需的 {model, token}，看不到 prompt 与回答正文。
  • 官方上游：OpenAI（Authorization: Bearer）、OpenRouter（Bearer + HTTP-Referer / X-Title）、Anthropic（x-api-key + anthropic-version）、Gemini / AI Studio（x-goog-api-key）。enclave 用编译期写死的官方地址加严格 TLS 直连，凭据在 enclave 内注入。

本方案不声称防护的范围（如实说明）：上游提供商拿到的仍是你的 prompt 明文——本方案保护的是「本网关及其宿主环境看不到你的数据」，而非上游模型服务方；SGX 侧信道等硬件层面的攻击不在本文讨论范围；单纯拿到 Quote 还不足以完全杜绝伪造，必须再做 DCAP 签名链验证（relay-verify -dcap-verify）。后续章节将逐层展开每一环的实现与验证方法。

2 · SGX Enclave / 机密计算

本系统的安全根基不是"我们承诺不看你的数据"，而是"我们在物理上看不到你的数据"。支撑它的硬件机制是 Intel SGX（Software Guard Extensions）：CPU 在自己内部划出一块受硬件保护的隔离区——enclave（飞地），其代码与数据驻留在一段被 CPU 实时加密的物理内存 EPC（Enclave Page Cache）里。明文只在 CPU 封装（package）内部——缓存、寄存器——短暂存在；一旦写回物理内存 DRAM 或流经内存总线，全部是密文。

关键在于谁持有解密密钥：EPC 的加密密钥由 CPU 在每次上电时随机生成，只存在于芯片内部，永不写入 DRAM、永不暴露给任何软件。因此无论攻击者拥有多高的软件特权（root、内核、hypervisor），还是掌握整台物理机（云厂商、运维、机房），他们能触及的都只是一段无意义的密文——特权不等于可读。这与传统隔离有本质区别：进程隔离、虚拟机隔离都依赖 OS / hypervisor 这个"更高特权层"来执行，而 OS 与 hypervisor 恰恰是 SGX 明确不信任的对象。

2.1 明文 vs 密文：同一台机器里的两块内存

下图对比同一台主机上两种内存的可读性。左边是普通进程内存：prompt、API Key、响应都以明文躺在 DRAM 里，任何特权角色一条 gdb / dd / 内存快照就能整块 dump。右边是 enclave 的 EPC：同样这三个角色伸手去读，只会拿到 CPU 实时加密后的密文，被挡在 CPU 加密边界之外；唯一能看到明文的，是运行在 enclave 内部、且被度量过的那段代码。

2.2 加密发生在哪里：MEE 与内存总线

"内存被 CPU 加密"的具体含义是：CPU 封装内的内存加密引擎（MEE，Memory Encryption Engine）坐在缓存与内存控制器之间。enclave 的数据写出 CPU 时被 MEE 加密并附加完整性校验（防篡改、防重放）；读回时先在 CPU 内解密、校验，才交给 enclave 代码。于是明文的整个生命周期被牢牢锁在 CPU 封装这条边界之内。

这带来两层防护：其一，软件层——OS / hypervisor 即便映射并读取 enclave 的物理页，看到的也只是密文；其二，物理层——总线探针、恶意 DMA、冷启动（cold-boot）攻击、乃至直接拔走内存条，截获到的同样是密文。

2.3 反转的信任模型：谁能读到明文

SGX 把传统信任链反转了：以往你必须信任 OS、hypervisor、云平台"不作恶"；在 SGX 下，这些层被移出可信基（TCB），可信基收缩到"CPU 硬件 + enclave 内那段可审计代码"。下表列出各角色对两类内存的可读性。

2.4 在本系统中的落地

我们把 AI 请求的热路径——relay-core——整体放进 enclave，用 Gramine-SGX 1.9 这个 LibOS 把 Go 二进制无改造地跑进 SGX。远程认证走 DCAP（Data Center Attestation Primitives）：CPU 出具的 quote 内嵌 MRENCLAVE——对 enclave 初始代码页与配置（manifest）算出的度量指纹。本系统当前的度量值为：

MRENCLAVE = 98ba342adb8092d60c940e75ce8e07036c96226595b03f16bb8d35a52a1872ee

RA-TLS 把这份硬件身份绑定到网络通道：enclave 在内部生成 TLS 密钥后，将 report_data = SHA-512(TLS SubjectPublicKeyInfo) 写进 quote，客户端据此确认"正在和我通话的这条 TLS，其私钥就活在被认证的 enclave 内部"（详见第 3、4 节）。

与之对照，new-api 是不可信的控制面（control plane），运行在 enclave 之外的普通宿主进程里。它只做元数据决策——用哪个渠道、扣多少 token——跨越信任边界流向它的只有 {model, token}，prompt 正文永不外泄。enclave 拿到路由决策后，直接对官方上游发起严格 TLS 连接：OpenAI（Bearer）、OpenRouter（Bearer + HTTP-Referer / X-Title）、Anthropic（x-api-key + anthropic-version）、Gemini / AI Studio（x-goog-api-key）。

为了让 MRENCLAVE 可被任何人独立复算，构建被固定为可复现构建（reproducible build）：锁定 Go 1.26.1 + Gramine 1.9 及基础镜像，两次编译得到字节一致的二进制，从而收敛到同一 MRENCLAVE——指纹因此可以脱离对发布者的信任而被验证。

2.5 诚实边界

SGX 不是绝对黑箱。EPC 的加密与完整性保护针对的是"读明文 / 篡改内存"这类威胁；历史上存在针对 SGX 的微架构侧信道（side-channel）研究（如缓存时序、瞬态执行类）。DCAP 的 TCB 状态检查能缓解"微码 / 平台过期"一类风险，但侧信道属于另一层持续演进的风险面。本节的所有结论都以"客户端确实完成了远程认证与度量比对"为前提——不验，则一切回到纯粹的信任。

3 · MRENCLAVE 代码度量

要证明"服务器跑的就是这份不落盘的审计代码"，首先得有一个无法伪造的代码身份。在 SGX 里，这个身份就是 MRENCLAVE——一个 32 字节（SHA-256）的度量值。它不是由运营方声明的，而是在 enclave 装载的一瞬间，由 CPU 硬件逐页计算出来的。运营方无法在不改变这个指纹的前提下，往 enclave 里塞进任何一行"偷偷写盘"的代码。

3.1 逐页度量：EADD → EEXTEND → EINIT

Gramine-SGX 1.9 把 relay-core 及其运行时打包成 enclave 镜像后，交给 CPU 装载。装载过程中，CPU 对每一页内存执行两条指令：EADD 把页加入 enclave，EEXTEND 把该页按 256 字节为单位喂进一个持续累积的 SHA-256 中。喂进哈希的不只是页里的字节，还包括这一页的虚拟地址（在 enclave 里的偏移）和安全属性（SECINFO：读/写/执行权限、页类型）。所有页处理完后，EINIT 冻结这个累积值——它就是 MRENCLAVE。

  • 代码页（.text）——可执行指令，即 relay-core 的全部业务逻辑。
  • 数据页（.data / .rodata）——初始化数据与常量，例如编译写死的官方上游 URL。
  • 堆 / 栈 / 控制结构（SSA、TCS）的页布局——不是内容，而是"哪一页放在哪个地址、给什么权限"。
  • manifest（relay-core.manifest）——Gramine 的信任配置：入口点、环境变量、可信文件哈希、挂载策略。它同样进入度量，所以改配置也会改指纹。

因为地址与权限都进了哈希，即使两份 enclave 的代码字节完全一样，只要页布局不同，MRENCLAVE 也会不同。这正是"逐页度量"比"只哈希一个二进制文件"更强的地方：它锁定的是加载进内存后的完整状态，而不只是磁盘上的文件。

3.2 雪崩效应：改一字节，指纹全变

SHA-256 具有雪崩效应（avalanche effect）：输入哪怕只翻转 1 个 bit，输出也会有约一半的 bit 随之翻转，且新旧值之间毫无可预测的关系。对 enclave 而言这意味着：运营方想在代码里插一句"把 prompt 写进日志"，或在 manifest 里悄悄挂载一个可写目录，得到的都将是一个与 pin 值毫不相干的新指纹——客户端在验证第 2 步（度量比对）会立刻变红，请求根本发不出去。

3.3 类比：你早就在用的"内容指纹"

MRENCLAVE 并不神秘——它和你日常依赖的多种"内容寻址"指纹是同一类东西：都用密码学哈希把一坨字节压成一个短标识，凭它就能判断"这份东西是不是那份确切的东西"。区别只在于 MRENCLAVE 由 CPU 在硬件里计算、并被 Intel 的远程认证 quote 签名背书，因此运营方连"谎报指纹"都做不到。

机制度量对象摘要 / 长度类比要点
MRENCLAVE（SGX） 代码页 + 数据页 + 页布局 + manifest SHA-256 / 32 字节 硬件度量、被 quote 签名，改一字节即失配
Docker 镜像 digest 镜像 config + 各 layer 内容 SHA-256（sha256:…） 内容寻址，pull 时按 digest 校验

把这三小节串起来：MRENCLAVE 由 CPU 逐页度量得到，被写进远程认证 quote 的 report body 并由 Intel 背书；客户端在验证时把它与自己 pin 的期望值 e5c301b5f4256e08…5bd9b36a 逐字节比对（见第 5 节验证流程的 Check 2）。而这个期望值之所以可信，是因为整套构建可复现——固定 Go 1.26.1 + Gramine 1.9，任何人独立重新编译都会得到同一个 MRENCLAVE。于是"运营方看不到你的 prompt"这句承诺，被压缩成了一个可以当场核对的 32 字节数字。

4 · DCAP Quote 结构剖析

客户端向 enclave 索取的 attestation 里，核心是一段名为 quote 的二进制。它不是自由格式的 JSON，而是一个由 Intel SGX ABI 规定的、little-endian、定长且带版本的结构体：前 48 字节是 Quote Header，紧接 384 字节的 Report Body（sgx_report_body_t），再往后是 ECDSA-P256 签名 与 QE 证书链（PCK）。正因为布局是硬件/固件定义的定长 ABI，验证器无需任何解析库，按写死的字节偏移裸切即可取字段——这也是为什么本系统的客户端只依赖两个偏移常量：112 与 368。

4.2 客户端只读这两段字节

控制面（new-api）与宿主机都无法伪造 quote，因此客户端的本地解析刻意极简：只切两段字节回答两个正交的问题——"跑的是不是那份代码"（MRENCLAVE）与"我是不是在直接和那个 enclave 通话"（REPORT_DATA）。其余字段（Header、ECDSA 签名、QE 证书链/PCK）客户端不解析，整体交给 DCAP（libsgx_dcap_quoteverify）上溯到 Intel SGX Root CA 验真。

  • MR_ENCLAVE @112（32 B）：CPU 在加载时对初始代码页 + 数据页 + manifest 算出的度量。可复现构建让客户端能独立重算出 pin 值 e5c3…9b36a，无需向运营方索取；只要相等，运行的字节就是那份审计过、不落盘、只连官方上游的代码。
  • REPORT_DATA @368（64 B）：enclave 在内部生成 TLS 密钥对时，把 SHA-512(SubjectPublicKeyInfo) 写进这 64 字节；它被 CPU 签进 quote，从而把"这把公钥"与"这个 enclave"硬件绑定，堵死 quote 回放 / 中间人代理。
  • 不由客户端解析的部分：Header、ECDSA-P256 签名、QE 证书链（PCK）——它们的价值在 DCAP 阶段体现，用来把 quote 的真伪上溯到 Intel SGX Root CA，并核对 TCB / QE identity。

5 · 认证签名链到 Intel 根 CA

第 4 节的纯 Python 验证器只做了三件"轻活":取回 quote、按固定偏移切出 MRENCLAVE 与 report_data、比对哈希。它把结论诚实地标成 PARTIAL PASS,因为它没有回答最关键的问题:凭什么这份 quote 是真的、而不是运营方自己编出来的一堆字节?一个能控制整台宿主机的攻击者,完全可以伪造一段结构合法、MRENCLAVE 与 report_data 都"对得上"的假 quote,骗过 Check 1–3。真正让伪造不可能的,是本节要讲的认证签名链——一条从 quote 一路上溯、最终锚定在 Intel SGX Root CA 的密码学签名链。

5.1 谁给 quote 签名:从 Quoting Enclave 到 PCK

DCAP(ECDSA)认证的签名不是 relay-core 自己盖的章。relay-core 只能通过 /dev/attestation/ 产出一份描述自身的本地 report(内含 MRENCLAVE 和写入的 report_data)。给这份 report 背书的是 Quoting Enclave(QE)——一个由 Intel 提供、并被 Intel 度量/签名的架构级 enclave。QE 用它的 attestation key 对 report 做 ECDSA 签名,而这把 attestation key 又由平台的 PCK(Provisioning Certification Key)私钥认证。简化成一句话:quote 最终被 PCK 私钥背书,而 PCK 私钥只由该 CPU 平台/Intel 供给持有,运营方拿不到、也复制不出。它不在 enclave 里、不在宿主上、更不在 new-api 主体里。

5.2 一路验到 Intel SGX Root CA(客户端 pin 的可信根)

PCK 私钥对应一张 PCK 证书,这张证书由 Intel 的中间 CA(SGX Platform / Processor CA)签发,中间 CA 又由 Intel SGX Root CA 签发。验证器里硬编码固化了 Intel SGX Root CA 的公钥(信任锚)。验证时逐级验签:quote 的签名要能被 PCK 证书验通,PCK 证书要能被中间 CA 验通,中间 CA 要能被 pinned 的 Root CA 验通——任何一级断裂,整条链失效,验证器立即拒绝。信任因此不落在任何运营方或缓存服务身上,而是落在一枚出厂即烧录、全球公开、客户端自己 pin 的 Intel 根公钥上。

5.3 PCCS 只提供 collateral,伪造不了签名

逐级验签需要一批"料":PCK 证书链、平台 TCB 等级信息(TCBInfo)、合法 QE 的身份度量(QEIdentity)、以及证书吊销列表(CRL)。这批料合称 collateral,由 PCCS(Provisioning Certification Caching Service) 缓存与分发(本系统用阿里云 PCCS,见 /etc/sgx_default_qcnl.conf)。关键在于:PCCS 里的每一份 collateral 都已经被 Intel 事先签过名,而 PCCS 本身没有任何 Intel 私钥。它只是个搬运工。所以即便 PCCS 恶意、被入侵、或到 PCCS 的链路被 MITM,它顶多能塞给你一份伪造/篡改的 collateral——而验证器会拿 pinned 的 Intel Root 公钥去验这批料的签名,伪造的东西当场验不过,直接拒绝连接。

collateral内容由谁签名恶意 PCCS 篡改后的后果
PCK 证书链平台供给密钥证书 + 中间 CAIntel SGX Root CA 体系验签不通过 → 拒绝
TCBInfo平台 TCB(微码 / SVN)等级与状态Intel(TCB Signing Cert)签名失效 / TCB 判定异常 → 拒绝

为什么恶意 PCCS 终究没用,可以归纳成四点:

  • 没有 Intel 私钥——它造不出任何新的、能被 pinned Root 验通的 PCK / 中间 / 根签名。
  • collateral 全是公开且可离线核验的——甚至能由 enclave 的 /attestation/collateral 端点带外一次性喂给验证器,完全绕开 PCCS。
  • 信任锚不是 PCCS——验证器 pin 的是 Intel Root 公钥,从不假设 PCCS 诚实。
  • 最坏是 fail-closed——PCCS 宕机 = 拿不到 collateral = 验证失败并拒连,而不是被骗放行。

这正好补齐了第 4 节客户端脚本 PARTIAL PASS 里缺的那一环:DCAP 签名链验证。有了它,quote 从一句"声称"变成一份由 Intel 硬件根 CA 背书、任何人都伪造不了的数学声明——而 MRENCLAVE 比对(代码就是那份不落盘的审计构建)与 report_data 通道绑定(防 MITM 转发)则在此基础之上,继续把结论钉死。

6 · RA-TLS 通道绑定(防转发 / 中间人)

第 5 节的 MRENCLAVE 比对只回答了半个问题:它证明"世界上某处存在一台跑着那份审计过、不落盘代码的真 enclave"。但它没有证明"此刻正在和你通话、终结你 TLS 连接的这个端点,就是那台 enclave"。缺了这一环,一个恶意运营方完全可以从别处一台诚实 enclave 抓一份合法 quote 回放给你,同时用一台普通的、能偷看明文的代理来终结你的 HTTPS——MRENCLAVE 照样能对上,你却把 prompt 明文交给了代理。

RA-TLS 通道绑定(channel binding)就是用来堵死这条转发路径的:让被硬件签名认证的那台 enclave,与正在承载这条 TLS 连接的端点,在密码学上锁死为同一个。做法只有一句话——把 enclave TLS 公钥的指纹写进 quote,客户端再用自己实际连上的证书去核对。

6.1 绑定是怎么建立与核验的

enclave 在接受任何请求之前,先在 SGX 加密内存(EPC)里现场生成一对 TLS 密钥(EC),私钥永不导出、永不落盘;对证书里的 SubjectPublicKeyInfo(SPKI,公钥结构)做 SHA-512 得到恰好 64 字节的指纹,写入 /dev/attestation/user_report_data,CPU 随即把它作为 quote 的 report_data 字段一并硬件签名。客户端从实际的 TLS 连接取回服务器出示的证书,同样计算 SHA-512(SPKI),再与从 quote 切出的 report_data(offset 368,长度 64B)逐字节比对——相等即证明:终结这条 TLS 的私钥就活在那台被认证的 enclave 内部,没有中间人。

6.2 为什么运营方无法把别处的合法 quote 转发过来冒充

关键在于:私钥无法离开 enclave。一份合法 quote 里的 report_data,绑定的是某一把特定公钥;要通过客户端的比对,冒充者必须让自己终结的 TLS 出示同一把公钥的证书,也就是必须掌握对应私钥。而 RA-TLS 的私钥只存在于生成它的那台 enclave 的加密内存里,任何代理、宿主 root、云平台都拿不到。于是转发攻击必然在指纹比对处露馅:

换言之,通道绑定把"quote 里那把公钥"和"你 TLS 实际用的那把公钥"钉在一起。冒充者要么拿不到私钥(转发别人的 quote,证书对不上),要么就得改代码在同一 enclave 内自己签——而那会改变 MRENCLAVE,在第 5 节的度量比对处红掉。两道检查互相咬合,构成"被认证的代码 = 正在通话的端点"这一完整结论。

要素值 / 计算由谁固定
report_dataSHA-512(TLS SubjectPublicKeyInfo),恰好 64 字节enclave 写入 user_report_data,CPU 签入 quote(offset 368)
服务器证书RA-TLS 自签,SPKI = 该 TLS 公钥enclave 内存现场生成,私钥不导出、不落盘

正因为 TLS 在 enclave 内部终止,你的 prompt 明文第一次出现就在 SGX 加密内存里;宿主机、运维、云平台即便在网卡抓包也只看到密文。作为不可信控制面的 new-api 只经手 {model, token} 这类元数据,永远碰不到正文——这正是"可验证 NO-LOG"链条中,由 RA-TLS 通道绑定负责钉牢的那一环。DCAP 对 quote 本身 Intel 签名链的验真(防止整体伪造一份 quote)见第 7 节;本节的通道绑定只解决"这份 quote 是不是属于正与我通话的端点"这一层。

7 · 可复现构建(指纹 ↔ 源码)

MRENCLAVE(见第 2 节)是一串 32 字节的 SHA-256 度量值,写在 DCAP quote 里。但一个度量值本身只回答"运行的是某份确定的字节",它并不告诉你那份字节对应哪份源码。没有可复现构建(reproducible build),MRENCLAVE 就只是一个数字:运营方完全可以从一份被篡改、会偷偷记录 prompt 的源码编译出另一个 enclave,照样出具一份合法的硬件签名 quote,而你无从把这个数字反查回"那份被审计、不落盘、只连官方上游"的源码(第 3、5 节)。可复现构建正是补上这一环的机制——它把"审计过的源码 == 正在运行的代码"从一句承诺变成任何人都能独立复核的等式。

7.1 没有可复现构建,MRENCLAVE 只是个数字

问题的核心是方向:度量是"源码 → 指纹"的单向哈希,而验证者拿到的只有指纹。若同一份指纹无法被任何人从公开源码重新推导出来,那么指纹与源码之间就不存在可核验的绑定——运营方声称"这个数字来自审计过的源码",而你只能选择相信。这正是我们要消除的那份信任。

7.2 确定性重编译:同一份源码 → 同一个 MRENCLAVE

补上这一环的办法是让构建确定性(deterministic):任何第三方拿公开源码(固定 git commit)按同一套配方重编译,都得到逐字节相同的 relaycore 二进制与 relaycore.manifest.sgx,于是 gramine-sgx-sign 算出的 MRENCLAVE 也相同。我们把编译器、构建标志、依赖、Gramine 版本、enclave 几何全部钉死,并放进一个 hermetic Docker 里渲染签名。下表是这套配方里每个"固定项",以及不固定它会怎样悄悄改变 MRENCLAVE。

7.3 MRENCLAVE 度量了什么(改一字节就全变)

MRENCLAVE 不是对二进制单独做哈希,而是 CPU 在 EINIT/EEXTEND 加载流程里,对 enclave 的初始状态做的增量 SHA-256:它一并折入了 relay-core 代码页、Gramine 加载器页、被度量的 manifest(内含每个 trusted-file 的哈希),以及内存几何(大小、线程数、页权限与顺序、debug 位)。所以 MRENCLAVE = 这份代码 + 这个 Gramine + 这份 manifest + 这套内存布局,任意一处改一个字节,指纹就整体翻新——这正是"指纹能代表源码"所需要的敏感性。

7.4 三方相等 ⇒ 无需信任发布者

确定性到手后,一个指纹就有了三个相互独立的来源:① 你(或任何第三方)照公开源码 + 公开配方自建得到的 MRENCLAVE_A;② 项目在签名 release 里公布、并用 GPG/cosign 把"MRENCLAVE ↔ commit"绑定的 MRENCLAVE_B;③ 线上服务实时出具的 DCAP quote 里携带的 MRENCLAVE_C。把三者放进同一个"全部相等?"判定:只要 A = B = C,由传递性即得"线上运行的,就是这份公开、可审计、不落盘、只连官方 URL 的源码"——而这一结论不需要相信发布者,因为你能自己跑第 ① 步复现出同一个数字。

这里有两个容易被忽视、却正是"无需信任"底气所在的性质:

  • 签名 key 与指纹无关。RSA 签名密钥只决定 MRSIGNER,不进入 MRENCLAVE 的度量。因此第三方复现时可以用自己的一次性 key 签名,依然得到与官方 release 完全相同的 MRENCLAVE——验证者永远不需要运营方的私钥,校验的是代码度量而非签名者。
  • 签名发布只解决"谁在断言映射",不替代复现。GPG/cosign 签名的 release 记录把"MRENCLAVE ↔ commit"绑定到一个可核验的身份上,方便他人对照;但真正的保证仍是那次独立重建——即使你完全不信任发布者的签名,只要自建出的 MRENCLAVE_A 等于线上 quote 的 MRENCLAVE_C,结论照样成立。

需要强调两条边界(详见第 8 节威胁模型):其一,可复现构建证明的是代码同一性,不是代码正确性——"这份源码值得信任"由第 3、5 节的审计给出,两者缺一不可;其二,生产 enclave 必须是非 debug(sgx.debug=false),debug enclave 允许宿主读写其内存,会彻底作废"不落盘"保证,且其 MRENCLAVE 与生产版不同,验证器须独立拒绝 debug 位。当"可复现 + 审计 + 三方相等 + debug=false"四者齐备,"运行的就是这份被审计的源码"便成为一条可以自己复核、无需信任发布者的等式。

8 · 无正文留存 —— 为什么"可度量"

市面上绝大多数"无日志（no-log）"承诺，本质是一个运行时开关：一条配置项、一段隐私条款、一份内部流程。你无法核实它，只能相信运营方"没有把日志打开"、"没有偷偷入库"。而本系统的"不存正文"是另一种东西 —— 它是 relay-core 二进制与 Gramine manifest 的结构性事实，并且这份结构被 MRENCLAVE（code + manifest 的度量值）固定下来，因此它是可度量、可验证的：任何人在发出 prompt 之前，都能用 DCAP quote 把 MRENCLAVE 比对一遍。

8.1 依赖闭包：能落盘的代码根本不在二进制里

对 enclave 入口执行 go list -deps ./cmd/relay-core，其一方（first-party）依赖闭包只有三个纯包—— pkg/officialurls（官方上游 URL 白名单）、pkg/relaycontrol（控制面 DTO 与客户端接口）、pkg/raenclave（RA-TLS / DCAP quote 生成）—— 外加 Go 标准库。这三个包都没有任何磁盘、数据库或日志副作用。new-api 里真正能落盘 / 写库 / 记日志的那些包（model 走 GORM 入库、common 含文件写入、logger 打日志到磁盘）一个都没有被 import。链接器（linker）因此从不会把这些代码拉进二进制。结果很干脆：能持久化正文的代码不在二进制里，运行时无论怎么调用都无法触发它。

8.2 文件系统：连"写的地方"都没有

再退一步做纵深防御（defense in depth）：就算某个 bug 真去调了 os.Create，Gramine manifest 的 fs.mounts 也没有给它任何落点。挂载表里只有 ——（1）一个 encrypted tmpfs 映射到 /tmp，它是 enclave 内的加密 EPC 内存、不由宿主承载、进程退出即焚；（2）只读的 trusted 二进制与 libs，全部度量进 MRENCLAVE；（3）一份 encrypted secrets 文件（密文存宿主，由绑定 MRENCLAVE 的密钥在 enclave 内解密）。没有任何可写宿主挂载，没有 allowed_files，也没有 encrypted-files 的宿主映射。所以最坏情况下，一次误写也只会落进"退出即焚"的加密内存，永远碰不到宿主磁盘。

8.3 承诺 vs 结构：这就是"可度量"的差别

把两种做法并排看，差别一目了然。传统 no-log 靠"运营方自律"，用户无从核实、也无从阻止一次单方的开关翻转；结构式 no-log 则把"不存正文"编码进了代码闭包与文件系统布局，任何改动都会改变 MRENCLAVE，从而在远程认证阶段当场失败。

维度承诺式 no-log（传统）结构式 no-log（本系统）
不存正文的机制 配置开关 / 隐私条款 / 内部流程 依赖闭包不含持久化代码 + manifest 无可写宿主路径
谁来保证 运营方自律，须相信其"没打开日志" 无需信任：代码结构 + 文件系统被 MRENCLAVE 度量

换句话说：本系统不请你"相信我们不记录"，而是把"记录正文"这条路径从二进制和文件系统里结构性地拿掉，再用一个你能亲自验证的 MRENCLAVE 把这份构造钉死。这就是"可度量"三个字的全部含义。

9 · 官方 URL 防 MITM + 多 provider 鉴权

第 8 节保证了"进 enclave 的只有 {model, token}"，但还有一个对称的问题：enclave 出去时到底连的是谁？如果运营方能把"官方 OpenAI 渠道"的目的地悄悄改成 evil.com，那明文 prompt 一出 enclave 就落进了中间人（MITM）手里，前面所有的度量与不落盘都白费了。本节的答案是：每个官方渠道的目的地 URL、以及每家上游的鉴权画像（auth profile），都被编译写死进 enclave、被 MRENCLAVE 度量；宿主 / new-api 给的 base_url 对官方渠道一律无视，enclave 只拨打自己代码里那份写死的官方 host。

9.1 目的地写死进 enclave：宿主的 base_url 被无视

官方 host 表放在一个纯叶子包 pkg/officialurls 里——它只依赖标准库，不拉入 model / service / logger 等任何业务包，因此能干净地链接进 relay-core enclave，并随代码一起被逐页度量进 MRENCLAVE（见第 3 节）。表按 channelType 下标索引，例如下标 1 就是 https://api.openai.com。因为这张表在 .rodata 里，改动其中任何一个字节都会让 MRENCLAVE 失配、客户端认证当场变红。

运行时，enclave 从不信任控制面回传的目的地。控制面在选路结果里带一个 is_official 字段，但那只是"提示（host hint）"；enclave 收到后用 officialurls.HasOfficial(channelType) + ProfileFor(channelType) 自己重新判定，再用 officialUpstreamURL() 把请求 URL 用"编译写死的 host + 客户端原始 path"拼出来。宿主传来的任何 base_url 覆盖值,在官方分支里根本不会被读取。这就是防 MITM 的核心：目的地由 enclave 里的代码决定，不可信的宿主动不了它。

9.2 每家一套鉴权画像（auth profile）

不同 provider 的鉴权方式并不一样，这些差异同样被写进 enclave、被度量。relay-core 保持"忠实透传"：客户端按各家原生格式发到原生 path，enclave 只负责强制官方 host、按下表注入标准凭据、逐字节转发 body、绝不落盘。注意——客户端进网关用的 token 永远不会被转发到上游；上游真正的 API key 由 enclave 自有的密钥库注入，宿主看不到。

Provider（type id）官方目的地 host鉴权头额外必需头
OpenAI（1） api.openai.com Authorization: Bearer <key> —
OpenRouter（20） openrouter.ai/api Authorization: Bearer <key> HTTP-Referer、X-Title（归因头）
Anthropic（14） api.anthropic.com x-api-key: <key> anthropic-version: 2023-06-01
Gemini / AI Studio（24） generativelanguage.googleapis.com x-goog-api-key: <key> —

这份 profile 表本身就是一份可审计的官方上游白名单：只有表里登记的 provider 才能被 enclave 当作"官方"来服务。远程验证者核对 MRENCLAVE，就等于确认了"这套 enclave 到底能以官方身份连哪几家、各自怎么注入凭据"——一切都是编译期确定、无法在运行时被运营方偷偷加塞或改写的。

9.3 enclave 自主重判：不信宿主的 is_official

把这套逻辑压成一句话：宿主只能"建议"，enclave 自己"裁决"。控制面回传的 is_official 是提示；enclave 用编译进自己的官方表重新判定——命中官方 URL 表且有对应鉴权画像，才用"写死 host + 该 provider 专属鉴权头"经 strict-TLS 外呼；否则一律拒绝为"非可信官方上游"（HTTP 400）。运营方既没法把官方指向中间人，也没法凭空把一个未登记的上游冒充成官方。

10 · 威胁模型与诚实边界

一套安全系统的价值，取决于它诚实地说清"在什么假设下、挡住了谁、又挡不住什么"。本系统的信任边界很干脆：可信计算基（TCB）只有三样东西——SGX 硬件（CPU 对 EPC 内存的透明加解密）、被逐页度量、其指纹等于 pin 值 e5c3…9b36a 的 relay-core enclave，以及客户端那份锚定在 Intel SGX Root CA 公钥上的验证逻辑。其余一切都在边界之外且默认不可信：运营方、宿主 OS、云 hypervisor、网络链路，全都可以任意作恶，却拿不到你的 prompt/response 明文。下面用对手能力矩阵把这条边界钉死，再用"诚实边界"一节讲清它不承诺什么。

10.1 对手能力矩阵

把每一类现实对手拆成两列——"能做什么"（它确实拥有的能力，我们不否认）与"不能做什么"（SGX + DCAP + RA-TLS 从密码学与硬件上封死的能力）。注意最后一行：官方上游本就是 prompt 明文的合法收件人，这不是漏洞，而是这套架构诚实标注的"必要信任"。

对手能做什么（承认的能力）不能做什么（被 TCB 封死）
恶意运营方
new-api 控制面 / 部署方		 决定路由与计费（{model, token}）；看到调用元数据（模型、token 计费数、时间）；选择部署哪份镜像；随时停机 / 拒绝服务 读取 prompt / response 明文；在不改变 MRENCLAVE 的前提下改动 enclave 代码或 manifest；伪造 quote（手里没有 Intel 私钥）；从 new-api 侧导出 enclave 内 TLS 私钥
宿主 OS / root

10.2 诚实边界：这套系统不承诺什么

上面的矩阵是"好消息"的一面。真正的安全承诺必须同样清楚地写下"坏消息"——那些前提条件与残余风险。任何回避下列几条的宣传，都应被视为不诚实：

  • SGX 微架构侧信道是已知弱点。Foreshadow（L1TF）、ÆPIC Leak、MDS/Plundervolt 一类的微架构侧信道，历史上确实能从 enclave 泄出秘密。缓解手段是及时更新 TCB（微码 + 平台固件），并在 DCAP 阶段用 TCBInfo / SGX SVN 下限拒绝过期平台。我们据实相告：可以把风险压到很低，但不能声称对侧信道 100% 免疫。
  • 客户端必须"真的验证"才有保证。这套体系的安全性完全落在客户端那几步验证上：验 DCAP 签名链、比对 MRENCLAVE、核对 report_data 通道绑定。跳过验证 = 退化成一条普通 HTTPS，运营方是否作恶你无从知晓；第 4 节脚本那种 PARTIAL PASS 也不等于安全，只是"结构对得上、但签名链未验"。
  • debug enclave 与生产 enclave 的 MRENCLAVE 不同。以 debug 模式（sgx.debug = true）加载的 enclave，其内存可被调试器直接读取，机密性形同虚设；而它的度量与生产 enclave 完全不同。因此生产部署必须 sgx.debug = false，且客户端 pin 的必须是生产 MRENCLAVE e5c3…9b36a——一旦有人拿 debug 镜像顶替，度量比对当场变红。
  • 纯结构 / 度量检查之外，还必须做 DCAP 签名链验证。只按偏移切出字节、比对 MRENCLAVE 与 report_data，无法阻止一个控制宿主机的攻击者伪造一份"字段都对得上"的假 quote。唯一能证伪它的，是把 quote 的 ECDSA 签名经 PCK 证书链一路验到客户端 pin 的 Intel SGX Root CA（见第 5 节）——缺了这一步，前面的比对都是空中楼阁。
  • 可复现构建是"源码 ↔ 指纹"的前提。你之所以敢 pin e5c3…9b36a，是因为固定 Go 1.26.1 + Gramine 1.9 的构建可被任何人独立复现，从公开源码重算出同一个 MRENCLAVE。若构建不可复现，这个指纹就无法被独立核对，你就只能盲信运营方"这份指纹对应那份源码"——那等于没有 attestation。
  • 元数据（token 数 / 模型 / 时序 / 长度）仍会泄漏。内容保密不等于流量分析保密。运营方与网络中间人看不到 prompt 正文，但依旧能观察到你调用了哪个模型、计费多少 token、在什么时间、请求与响应的字节长度与节奏。想进一步降低这类侧信息，需要 padding、批处理、固定节奏等额外手段——本系统不默认提供，也据实说明。

把两节合起来看：本系统的承诺是有条件的、且条件全部可核验——当且仅当客户端真的验证了完整签名链、生产 enclave 关闭了 debug、构建可复现、平台 TCB 保持更新时，"运营方 / 宿主 / 云 / 网络看不到你的 prompt"这句话才成立；即便如此，元数据仍会泄漏，侧信道仍是需要持续对抗的残余风险。把这些边界摊开讲清楚，本身就是这套安全姿态的一部分。

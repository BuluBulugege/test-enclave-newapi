客户端验证器 verify_enclave.py — 为什么它能"证明"（而不只是"检查"）

一个前端/后端工程师读这份脚本，第一眼会觉得它只是"用 Python 发个 HTTPS 请求、切几段字节、比几个哈希"——普通到不能再普通。真正的分量不在代码复杂度，而在于它把一个信任问题翻译成了数学 + 硬件问题：传统上你要"相信运营方不看、不存你的 prompt"；这份脚本把这份信任整体搬走，落到两样你不需要相信任何人就能独立核验的东西上——密码学哈希（指纹相等，SHA-256 / SHA-512）与 Intel SGX 硬件签名（CPU 出具、Intel 背书的 quote）。运营方即便是 root、即便控制整台宿主机与云平台，也无法在不改变指纹的前提下伪造这三件事。信任因此从"相信运营方"转移到"数学 + Intel 硬件"。

0 · 先立心智模型：不是"看服务器说了什么"，而是"逼硬件签字画押"

整份脚本围绕一个对象——quote（远程认证凭证）：SGX CPU 生成的一段二进制，里面写死了两样关键字段， MRENCLAVE（当前enclave里那份代码+配置的硬件度量指纹）与 report_data（enclave 自己塞进去的 64 字节，用来绑定它的 TLS 公钥）。

脚本要做的三步：① 把 quote 取回来并切出这两个字段；② 用 MRENCLAVE 回答"跑的是不是那份审计过、不落盘的代码"；③ 用 report_data 回答"被认证的 enclave 是不是此刻正和我通话的这个端点"。真正让 quote 无法伪造的 Intel 签名链，交给 DCAP（见最后一节）。

1 · unverified_ctx() — 为什么故意关掉 CA / 主机名校验

def unverified_ctx():
    # We deliberately do NOT verify the TLS chain against a CA: this is an
    # RA-TLS self-signed certificate. Trust is established by the SGX QUOTE, not
    # by PKI. (Same reason `curl -k` is correct here.)
    ctx = ssl._create_unverified_context()
    ctx.check_hostname = False
    ctx.verify_mode = ssl.CERT_NONE
    return ctx

这段在干嘛：造一个关掉证书链校验、关掉主机名校验（CERT_NONE）的 TLS 上下文，等价于 curl -k。后面所有连接都用它。

为什么这能"证明"（而不是自毁安全）：普通 HTTPS 的信任根是 CA/PKI——某个证书颁发机构替证书背书。可这里 enclave 出示的是一张 RA-TLS 自签名证书，没有 CA 会签它；用 PKI 去验只会永远失败，而且就算验过也证明不了任何关于 enclave 的事。这里的信任根本不是 CA，而是 SGX quote。所以我们主动把 PKI 关掉，避免误把无关的信任源当成保障。

关键：关掉 CA 校验没有削弱安全，因为证书的真伪改由 Check 3（通道绑定）用硬件重新确立——把服务器实际出示的这张证书的公钥指纹，去和 quote 里硬件签过的 report_data 对齐。类比前端：你不再相信地址栏那把"锁"图标，而是准备亲手对这把钥匙取指纹，再和一份硬件签名的声明比对。

2 · fetch_attestation() — 取回 quote，先判 attested

def fetch_attestation(host, port):
    url = f"https://{host}:{port}/attestation"
    req = urllib.request.Request(url, headers={"Accept": "application/json"})
    with urllib.request.urlopen(req, timeout=20, context=unverified_ctx()) as r:
        body = r.read()
    doc = json.loads(body)
    if not doc.get("attested"):
        raise RuntimeError(
            "endpoint reports attested=false: it is NOT running inside an SGX "
            "enclave (reason: %s)" % doc.get("reason", "unknown"))
    quote = base64.b64decode(doc["quote_b64"])
    report_data_hex = doc.get("report_data_hex", "")
    return quote, report_data_hex

这段在干嘛：对该端点的 /attestation 发 GET，拿到 JSON；先检查 attested 为真，再把 quote_b64 从 base64 解成原始二进制字节（report_data_hex 仅作参考展示）。

为什么这能证明：attested=false 意味着这个进程根本没在 enclave 里（比如模拟模式或普通进程）——此时直接抛错中止，因为后续一切都建立在"存在真 quote"之上。这一步同时提供活性（liveness）：quote 是此刻、从这个端点、经这条连接实时取回的，不是一段陈年缓存，这对 Check 3 的通道绑定至关重要。

注意此刻 quote 仍只是"声称"——它的真伪（是否由 Intel 硬件真签过）尚未证明，那属于 DCAP。但我们已经拿到了需要解剖的字节。

3 · parse_quote() — 为什么偏移 112/368、长度 32/64

QUOTE_HEADER_LEN = 48
MRENCLAVE_OFFSET = QUOTE_HEADER_LEN + 64   # report body starts at 48; mr_enclave at +64
MRENCLAVE_LEN = 32
REPORTDATA_OFFSET = QUOTE_HEADER_LEN + 320  # report_data at +320 in the body
REPORTDATA_LEN = 64

def parse_quote(quote):
    if len(quote) < REPORTDATA_OFFSET + REPORTDATA_LEN:
        raise RuntimeError("quote too short (%d bytes); malformed" % len(quote))
    mrenclave = quote[MRENCLAVE_OFFSET:MRENCLAVE_OFFSET + MRENCLAVE_LEN]
    report_data = quote[REPORTDATA_OFFSET:REPORTDATA_OFFSET + REPORTDATA_LEN]
    return mrenclave, report_data

这段在干嘛：quote 是一个定长二进制结构体。这里不引任何解析库，直接按写死的字节偏移切两段：MRENCLAVE = quote[112:144]、report_data = quote[368:432]；开头的长度校验挡掉被截断的畸形 quote。

为什么这些偏移是对的：这些数字不是魔法常量，而是 Intel SGX ABI 规定的、little-endian、稳定且带版本的布局：

SGX ECDSA Quote 字节布局 (little-endian)
+-----------------------------------------+----------+
| Quote Header                            |   48 B   |  版本 / attestation key type / QE 信息
+-----------------------------------------+----------+
| Report Body  (sgx_report_body_t, 384 B)            |
|   +0    cpu_svn / misc_select / attributes ...     |
|   +64   MR_ENCLAVE                      |   32 B   |  -> 绝对偏移 48 + 64  = 112
|   +128  MR_SIGNER                       |   32 B   |
|   +320  REPORT_DATA                     |   64 B   |  -> 绝对偏移 48 + 320 = 368
+-----------------------------------------+----------+
| Signature (ECDSA-P256) + QE Cert Chain             |  <- DCAP 用它验 Intel 签名链
+----------------------------------------------------+

Report Body 紧跟在 48 字节头之后，所以 body 内偏移 +64 的 mr_enclave 绝对偏移是 48+64=112，body 内 +320 的 report_data 是 48+320=368；长度 32/64 就是结构体里这两个字段的固定大小。因为布局是 Intel 定义的定长 ABI，裸切字节就够用，无需解析器。（尾部的 Signature + QE 证书链正是 DCAP 要验的部分，本脚本不碰。）

这段在干嘛：把切出来的 MRENCLAVE 转成十六进制，和 want_mre（你用 --mrenclave 传入、来自签名发布版或你自己可复现重建的值）逐字符比。不等就退出 1，直白地写"不要信任它"。

为什么这能证明：MRENCLAVE 是 CPU 在加载 enclave 时，对其初始代码页 + 数据页 + 配置算出的度量哈希（measurement，硬件强制、软件伪造不了，类比 Docker 镜像 digest / git commit，但你改不了它）。指纹相等之所以有意义，靠两个前提叠加：

前提① 可复现构建（reproducible build）：enclave 里的代码是开源的，任何人重新编译都得到同一个 MRENCLAVE。于是 want_mre 你能独立算出，不必向运营方索取。

前提② 审计：读这份源码可确认它没有任何把正文写盘/入库/记日志的代码路径，且只连编译期写死的官方上游 URL。

于是：MRENCLAVE(运行中) == MRENCLAVE(你审计过的构建) ⇒ 由传递性，此刻在 enclave 里执行的字节，就是那份审计过的构建。运营方哪怕只改 1 个字节去偷偷加日志，度量立即改变，这项检查当场变红。"不落盘"由此从口头承诺变成可验证。

5 · get_peer_cert_der + spki_sha512 + Check 3 — 通道绑定为什么能防转发/中间人

def get_peer_cert_der(host, port):
    ctx = unverified_ctx()
    with ssl.create_connection((host, int(port)), timeout=20) as sock:
        with ctx.wrap_socket(sock, server_hostname=host) as ssock:
            return ssock.getpeercert(binary_form=True)


def spki_sha512(cert_der):
    """SHA-512 of the certificate's SubjectPublicKeyInfo (matches how the
    enclave binds its TLS key into the quote). Needs the `cryptography` pkg."""
    try:
        from cryptography import x509
        from cryptography.hazmat.primitives import serialization
    except ImportError:
        return None
    cert = x509.load_der_x509_certificate(cert_der)
    spki = cert.public_key().public_bytes(
        serialization.Encoding.DER,
        serialization.PublicFormat.SubjectPublicKeyInfo,
    )
    return hashlib.sha512(spki).digest()

        print(f"    → possible relay/MITM: this quote may belong to a different enclave.")
        sys.exit(1)

这段在干嘛：再开一条到同一个 host:port 的 TLS，用 getpeercert(binary_form=True) 抓下服务器实际出示的证书（DER）。spki_sha512 取出证书里的 SubjectPublicKeyInfo（SPKI，公钥结构）并做 SHA-512。Check 3 要求：SHA-512(服务器 SPKI) == report_data（就是我们从 quote 里切出的那 64 字节）。

为什么这能防转发/中间人：光有 Check 2，只证明了"世界上某处存在一台跑着正确代码的真 enclave"——恶意运营方完全可以从别处一台诚实 enclave 抓一份真 quote 回放给你，同时用一台普通的、能偷看的代理来终结你的 TLS。通道绑定堵死这条路：

enclave 在内部生成 TLS 密钥对时，就把 SHA-512(自己的 SPKI) 写进了 report_data，而 report_data 是 CPU 签进 quote 的一部分。于是 quote 以硬件签名的形式声明："持有这把公钥的 enclave，正是跑这段代码的 enclave。"如果你此刻通话的端点出示的证书，其 SPKI 哈希正好等于这个值，那么终结你 TLS 的私钥就活在被认证的 enclave 内部——任何中间代理都拿不到那把私钥（enclave 私钥无法导出）。不等 ⇒ 有人在转发一份属于别的 key / 别的 enclave 的 quote ⇒ relay/MITM ⇒ 中止。

绑的是 SPKI（只含公钥身份），不是整张证书——所以有效期、序列号等外观字段变了都无所谓，被硬件锁定的只是"钥匙身份"。缺 cryptography 库时该函数返回 None，Check 3 被跳过并明确提示——此时结论强度下降，务必装库补上。

6 · 诚实边界：这份脚本没做什么，以及为什么"必须真的验"

    ./relay-verify -addr <host:port> -mrenclave <hex> -dcap-verify

脚本自己把结论标成 PARTIAL PASS，并明写"The quote's Intel signature chain was NOT verified by this script."——这是刻意的诚实。它的边界：

① 没验 Intel 签名链（最关键）。纯 Python 只切字节、比哈希，从没校验 quote 的 ECDSA 签名是否上溯到 Intel 背书的认证密钥。一份整体伪造的 quote（攻击者自己编字节，凑出对的 MRENCLAVE、以及匹配他自己 key 的 report_data）会骗过 Check 1–3。真正让伪造不可能的是 DCAP：libsgx_dcap_quoteverify 会验 quote 签名 → QE 证书链 → Intel SGX Root CA，并核对 TCB / QE identity（吊销状态、微码是否最新）。所以必须再跑：relay-verify -dcap-verify。

② MRENCLAVE 的意义 = 你 pin 的值背后的可复现构建 + 审计。若 want_mre 是运营方随口给的、或构建其实不可复现，Check 2 就什么都证明不了。这个 pin 必须你独立推导。

③ SGX 并非万能。侧信道 / 微架构攻击、TCB / 微码过期属于另一层风险面；DCAP 的 TCB 状态检查能缓解"平台过时"这一类，但历史上确有 SGX 侧信道研究，别把它当绝对黑箱。

④ 必须真的运行验证。把脚本当摆设、无视红色结果、或"验证通过后又偷偷换端点"，保证都会失效。结论只对验证当刻、这条通道、这个 quote 成立。

所以：无需信任运营方，只需信任数学 + Intel 硬件

这四件事是一条缺一环就不成立的链：quote 经 DCAP 验真（Intel 硬件确实签过它、没被伪造）+ MRENCLAVE 与你独立重建的指纹匹配（跑的确是那份开源、审计过、不落盘、只连官方 URL 的代码）+ report_data == SHA-512(服务器 SPKI) 的通道绑定（被认证的 enclave 就是此刻和你通话的端点，没有中间人转发）+ 你真的把这三项验了。四环齐备时，"运营方会不会偷看 / 记录你的 prompt"这个问题不再靠信任来回答——它在密码学与 Intel 硬件层面被否定：运营方想作恶就必须改代码，一改指纹立刻变，你的验证当场变红。于是你无需信任运营方，只需信任公开可复现的密码学，以及你手里这份自己能读懂的验证器。

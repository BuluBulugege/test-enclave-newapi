可验证的无日志 AI 网关

让"我们不看、不存你的 prompt"从一句承诺，变成用户在发送前就能亲自用密码学验证的数学事实。基于 Intel SGX 机密计算 + 远程认证，构建在开源 AI 网关 new-api 之上。

01问题

今天所有 AI 网关 / 代理都能看见并记录你的每一条 prompt——你只能选择相信运营方。

信任无法验证

企业、医疗、金融、法律、政务、Web3 用户把敏感数据发给 AI 网关时，"我们不记录日志"只是一句市场话术。没有任何技术手段让用户验证服务器真的没读、没存、没转卖。

运营方可随时作恶或被攻破

网关持有明文 prompt 和上游 API Key，可被内部人员、被入侵者、被合规传票、被恶意配置（把"官方渠道"偷偷指向中间人）获取。合规与隐私风险无法量化。

02方案

把 AI 请求的"请求→响应"热路径放进硬件加密的 SGX enclave；用户用远程认证，在发送任何 prompt 之前，先验证正在运行的代码。

信任从"相信运营方"转移到"相信 Intel 硬件签名 + 一份任何人都能重新编译比对的开源代码指纹"。运营方即使拿到 root、即使想作恶，也无处下手——一改代码，指纹（MRENCLAVE）就变，客户端当场拒绝。

① 不存正文

请求/响应正文全程只在 enclave 的加密内存里，绝不落盘、不入库、不进日志。可度量、可证明。

② 官方防 MITM

标记为"官方直连"的渠道，目的地 URL 编译进 enclave 并被度量；运营方无法把它偷偷改到中间人代理。

③ 上游 Key 可不落宿主

上游 API Key 可用 SGX 硬件封印，只有该 enclave 能解密，宿主只见密文。

03工作原理

客户端连的是 enclave，不是网关主体；网关主体只当"元数据顾问"，永远看不到正文。

客户端发 prompt 前先验 quote
▶
SGX Enclave（relay-core）RA-TLS 终止 · 加密内存转发 · 不落盘
▶
官方上游api.openai.com（严格 TLS）
▲
new-api 主体（不可信 / 可闭源）只回答"用哪个渠道 / 扣多少 token"——只见元数据，永不见 prompt

虚线框 = 信任边界之外。客户端"验 quote"这一步本身就证明了它连的是真 enclave 而非网关主体——因为网关主体造不出有效的 Intel 签名 quote。

证明链（为什么结论被"逼"出来）

环节做什么保证
远程认证 quoteSGX CPU 产出一份描述"正在运行的确切代码"的签名声明链到 Intel 根 CA，运营方伪造不了
MRENCLAVE 比对quote 里的代码指纹 = 你重新编译开源代码得到的指纹运行的就是那份"无落盘 / 官方 URL"的审计代码
通道绑定quote 内嵌 enclave TLS 公钥的哈希你连的这条 TLS 就是被认证的 enclave（防转发）
可复现构建任何人按源码重建得到同一 MRENCLAVE指纹 ↔ 源码，无需信任发布者

04现场验证（任何人可跑）

这不是幻灯片。下面的端点跑在真实的 Intel SGX 硬件上，任何人都能独立验证。

公开可验证端点

端点：https://8.217.148.82:8443

期望 MRENCLAVE：
e5c301b5f4256e086d9ea074e72da445d10a48323e5bc6139992ef995bd9b36a

开源仓库：github.com/BuluBulugege/test-enclave-newapi

三步验证（发 prompt 之前）

  • Check 1 取回 quote（描述运行中的确切代码）
  • Check 2 MRENCLAVE == 你独立重建得到的指纹
  • Check 3 quote 绑定到这条 TLS 通道（防转发）
  • + DCAP 链 验证 quote 由 Intel 签名（杜绝伪造）

用 Python 验证器

# 纯标准库即可跑 Check 1/2；Check 3 需 pip install cryptography
python3 verify_enclave.py --addr 8.217.148.82:8443 \
  --mrenclave e5c301b5f4256e086d9ea074e72da445d10a48323e5bc6139992ef995bd9b36a

真实输出（对上线端点跑出）

Verifying enclave at 8.217.148.82:8443

✓ Check 1 quote obtained (4734 bytes) and parsed
✓ Check 2 MRENCLAVE matches the pinned value
    e5c301b5f4256e086d9ea074e72da445d10a48323e5bc6139992ef995bd9b36a
✓ Check 3 quote is bound to THIS TLS channel (report_data == SHA-512(server pubkey))

RESULT: PARTIAL PASS (structure + MRENCLAVE + channel binding)

反例也成立：故意传入错误的 MRENCLAVE，验证器立即 ✗ Check 2 FAILED — MRENCLAVE MISMATCH 并拒绝。伪造的 quote 会在 DCAP 链验证处被拒。

05信任模型

诚实地划出：什么被信任、什么明确在信任之外。

组件是否在可信基（TCB）内
Intel SGX 硬件 + 微码、DCAP 认证是（根信任）
relay-core enclave 代码（开源 + 可复现）是（可审计，~1100 行）
客户端验证器是（用户自持 / 可审计）
new-api 主体、宿主 OS、数据库、运营方否（明确不可信）
云厂商 / hypervisor否（看不到 enclave 内存明文）

06市场与时机

AI 隐私焦虑达到顶点，机密计算刚好成熟——两条曲线在此交汇。

目标客户

受监管行业的 AI 使用：医疗、金融、法律、政务、跨境数据、以及 Web3 / 加密原生场景。

为什么是现在

SGX/TDX 已在 Azure、GCP、阿里云普及；远程认证工具链成熟；企业对"AI 会不会偷看我的数据"的顾虑空前强烈。

护城河

可复现构建 + 链上可发布的度量值 + 与 40+ 上游 provider 的 new-api 生态整合；"可验证"本身即差异化。

商业化路径：面向企业的"可证明无日志"AI 网关（私有部署 / SaaS）、可验证推理审计、以及与 Web3 结算结合的去信任 AI 市场。

07现状与诚实的边界

我们只讲能被验证的事实，并明确标出尚未闭合的部分。

已实现

  • relay-core 在真实生产 enclave 内运行（sgx.debug=false），产出真 DCAP quote
  • 公网端点 + Python/Go 客户端验证器（含 DCAP 签名链验证）
  • 无落盘正文的 CI 门禁 + 只挂加密内存的 manifest
  • 可复现构建：核心 Go 二进制两次编译字节一致、零外部依赖
  • 与 new-api 真实集成：真实用户 / 真实令牌 / 官方渠道，端到端打到真 OpenAI

尚未闭合

  • 研究原型，尚未经第三方安全审计
  • 客户端必须真的验证才有保证；SGX 存在已知的微架构侧信道研究性弱点
  • canonical 可复现构建需固定基础镜像 digest 与系统库版本
  • 本演示出口在受 OpenAI 地域限制的区域（不影响链路正确性）
